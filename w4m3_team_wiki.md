## 2. 주요 인사이트 

### 1. 네트워크가 아닌 'HTML 파싱'이 최대 병목

* **현상:** 웹 크롤러는 네트워크 I/O나 DNS 조회가 주요 병목일 것으로 예상했으나, 실제로는 **HTML 파싱이 최대 병목**이었다.
* **원인:** 웹페이지 평균 용량이 크게 증가했다. 
* **해결:** C++ 기반 고성능 HTML 파싱 라이브러리를 도입하고 본문을 **250KB로 자른(Truncate)** 후 파싱하여 속도를 대폭 개선했다.

### 2. CPU를 점유하는 SSL Handshake

* 데이터센터의 네트워크 대역폭은 충분했으나, 전체 웹 트래픽의 80% 이상이 HTTPS를 사용했다.
* **SSL 암호화/복호화 과정이 전체 CPU 시간의 25%를 점유**하면서 CPU 성능 한계로 인해 패칭속도가 제약되었다.

### 3. 여전히 유효한 순수 HTML 크롤링

* SPA(Single Page Application) 확산으로 JavaScript 실행 없이는 크롤링이 불가능할 것으로 보였으나, 여전히 수많은 웹 페이지가 전통적인 HTML 링크 구조를 유지하고 있어 JS 렌더링 없이도 10억 개 수집이 가능했다.

## 3. 아키텍처 및 시스템 구조

단일 대형 노드 방식 대신 **"독립 노드 수평 확장(Horizontal Scaling)"** 구조를 채택했다.


* **노드 간 통신 제거 (No Cross-Node Communication):** 도메인 시드를 12개 노드에 분배했다. 노드 간 네트워크 간섭 없이 완전 독립 동작하도록 설계했다.
* **로컬 NVMe 인스턴스 스토리지 활용:** S3 중앙 저장소 사용 시 발생하는 막대한 API/저장 비용을 회피하고 고속 NVMe 스토리지를 직접 활용했다.

## 5. 팀원별 논의 및 소감

* "웹 크롤러라고 하면 네트워크 속도가 전부일 줄 알았는데, SSL 암호화 연산과 HTML 용량 증가에 따른 CPU 파싱 병목이 가장 크다는 점이 매우 흥미로웠다."
* "S3 같은 클라우드 표준 서비스가 무조건 정답은 아니라는 걸 배웠다. 데이터 양과 API 호출 횟수에 따른 비용 구조를 사전 계산하여 NVMe 스토리지를 선택한 결정이 흥미로웠다."
* "노드 간 통신을 없애기 위해 도메인을 사전 샤딩하여 시스템을 단순화하고, 장애 발생 시 복구가 쉬운 Fault Tolerance 구조를 설계한 점이 인상적이었다."
