
# [2026-07-17] 회고

---

## Review (리뷰)
 - 실시간 배달 주문 시스템을 데이터 모델링 프로젝트의 주제로 선정하였다
 - 화이트 보드에 ERD를 직접 그리면서 필요한 엔티티와 속성에 대해 논의하였고 각 관계에 대해서 토론하였다
 - 개념 모델링과 논리 모델링 과정을 진행하면서 각 관계의 기수성과 필수성에 대해서 정리하였고 이를 기반으로 ERD를 그렸다
 - 물리 모델링 과정을 통해 각 속성명을 컬럼명으로 변환하였고, 다른 파트를 맡은 팀원들과 논의를 거쳐 용어의 표준화를 진행하였다
 - 각 속성의 타입과 제약조건을 정의하고 인덱스를 설정하여 물리 모델링 단계를 마무리하였다
 - 데이터 모델링을 진행하면서 중점적으로 보았던 부분을 발표자료로 만들어 발표를 준비하였다.

## 잘한 점
 - 팀원들과 화이트보드를 활용해 활발히 소통하며, 자칫 헷갈리기 쉬운 엔티티 간의 기수성과 필수성을 초반에 명확히 합의하고 넘어간 점이 훌륭했다.
 - 물리 모델링 단계에서 각자 맡은 파트만 분리해서 작업하지 않고, 함께 논의하며 용어 표준화를 진행해 데이터베이스 전체의 일관성과 퀄리티를 높였다.
 - 단순히 테이블을 쪼개는 것에 그치지 않고, 향후 실무에서의 조회 성능을 고려해 인덱스까지 선제적으로 고민하고 설계에 반영한 점이 뿌듯하다.
 - 모델링 결과물만 나열하는 발표가 아니라, '우리가 어떤 부분을 가장 깊게 고민했는지' 설계의 의도와 과정을 중점적으로 담아내어 설득력 있는 발표 자료를 구성했다.

## 아쉬운 점
 - '실시간 배달 주문'이라는 복잡한 도메인 특성상, 주문 상태 변화나 배달 이력 등 까다로운 관계를 어떻게 효율적으로 풀어나갈지 고민하느라 예상보다 설계 단계에서 시간이 많이 지체되었다.
 - 용어 표준화를 진행할 때, 프로젝트 극초반에 명확한 명명 규칙 가이드를 먼저 세워두지 않아 나중에 컬럼명을 일괄 수정해야 하는 번거로움이 있었다.
 - 설계는 꼼꼼히 마쳤지만, 우리가 설정한 인덱스와 데이터 타입이 실제 대용량 트래픽 쿼리 상황에서 어느 정도의 성능을 낼지 직접 테스트해 보지 못한 점이 아쉽다.
