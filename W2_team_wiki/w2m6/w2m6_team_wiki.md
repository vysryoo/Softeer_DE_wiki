## 1. Docker를 사용하는 이유가 뭘까요? (장점 및 도입 배경)

팀원들이 생각한 Docker의 장점은 다음과 같았습니다.

1. **가볍고 실행 속도가 빠르다.**  
   Docker는 기존 VM처럼 게스트 OS를 따로 설치하지 않고 호스트 OS의 커널을 공유하기 때문에 비교적 리소스 소모가 적고 실행 속도가 빠릅니다.

2. **실행 환경을 코드처럼 관리할 수 있다.**  
   Dockerfile을 통해 필요한 라이브러리, 실행 환경, 파일 복사 과정 등을 명시할 수 있습니다. 따라서 같은 환경을 반복해서 만들기 쉽고, 환경 설정 과정을 문서처럼 남길 수 있습니다.

3. **이미지 기반으로 배포와 롤백이 쉽다.**  
   한 번 만든 Docker image를 ECR 같은 저장소에 올려두면, 다른 서버에서도 같은 이미지를 pull 받아 실행할 수 있습니다. 문제가 생겼을 때는 이전 버전의 이미지로 되돌리는 것도 가능합니다.

4. **개발 환경과 배포 환경의 차이를 줄일 수 있다.**  
   로컬에서 실행하던 환경을 이미지로 묶어두면, EC2 같은 서버에서도 동일한 환경으로 실행할 수 있습니다.

이 중에서 팀이 가장 중요하게 본 장점은 **환경의 재현성**입니다.

이번 미션에서는 JupyterLab, Python 라이브러리, W1/W2 노트북, 데이터 파일을 하나의 Docker image로 묶었습니다. 그 결과 로컬 Mac에서 실행하던 환경을 AWS EC2에서도 동일하게 실행할 수 있었습니다.

결론적으로 Docker를 사용하는 가장 큰 이유는 **개발 환경과 배포 환경의 차이를 줄이고, 어디서든 같은 실행 환경을 재현할 수 있기 때문**이라고 정리할 수 있습니다. 즉, "제 컴퓨터에서는 되는데요" 문제를 줄이는 것이 Docker의 가장 큰 장점이라고 보았습니다.

## 2. 어떤 점은 더 불편한가요? (단점 및 한계)

팀원들이 생각한 Docker의 불편한 점은 다음과 같았습니다.

1. **초기 학습 부담이 있다.**  
   Dockerfile 작성, image build, tag 지정, container 실행, ECR push/pull 등 처음에는 익혀야 할 개념과 명령어가 많습니다.

2. **수정사항 반영 과정이 번거롭다.**  
   노트북이나 코드, 데이터 경로를 조금 수정해도 EC2에 바로 반영되지 않습니다. 수정 후에는 Docker image를 다시 build하고, ECR에 push한 뒤, EC2에서 pull/run을 다시 해야 합니다.

3. **이미지 용량이 커질 수 있다.**  
   실행 환경, 라이브러리, 노트북, 데이터 파일이 이미지에 포함되면 이미지 크기가 커질 수 있습니다. 이 경우 build, push, pull 시간이 길어질 수 있습니다.

4. **여러 컨테이너 운영 시 관리가 복잡해진다.**  
   컨테이너가 많아지면 단순히 `docker run`으로 관리하기 어렵고, 모니터링, 스케일링, 장애 복구를 위해 ECS, EKS, Kubernetes 같은 오케스트레이션 도구가 필요해집니다.

이 중에서 팀이 가장 크게 느낀 단점은 **수정사항 반영 과정이 번거롭다는 점**입니다.

이번 미션에서도 노트북이나 데이터 경로를 수정한 뒤 EC2에 바로 반영되는 것이 아니라, 로컬에서 Docker image를 다시 build하고, ECR에 push한 다음, EC2에서 기존 컨테이너를 중지/삭제하고 새 이미지를 pull/run 해야 했습니다.

결론적으로 Docker의 가장 큰 단점은 **실행 환경을 안정적으로 고정하는 대신, 수정과 재배포 과정이 복잡해진다는 점**이라고 볼 수 있습니다. 작은 수정도 이미지 재빌드와 재배포 과정을 거쳐야 하기 때문에 개발 초기에는 오히려 번거롭게 느껴질 수 있습니다.


## 3. 이번 미션에서는 하나의 EC2에 하나의 Docker container를 배포했습니다. 만약에 여러대의 EC2에 여러 개의 컨테이너를 배포해야 한다면 어떻게 해야 할까요?

* **컨테이너 오케스트레이션 도구 도입:**
  * Kubernetes, Amazon ECS/EKS 같은 오케스트레이션 도구를 사용해 여러 EC2(노드)에 컨테이너 배치, 스케일링, 장애 복구를 자동화합니다.
* **CI/CD 파이프라인 기반 배포 자동화:**
  * GitHub Actions, Jenkins 등의 도구를 활용하여 소스 코드 변경 시 자동으로 Docker 이미지를 빌드하고 AWS ECR 등의 컨테이너 레지스트리에 저장한 뒤, 오케스트레이션 도구와 연동하여 여러 EC2 인스턴스에 무중단으로 자동 배포되도록 파이프라인을 구축합니다.
