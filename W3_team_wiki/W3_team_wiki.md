# W3

## W3M2b: XML 파일 분석 및 위키 정리

### 분석 내역

**팀원 A: 정기준**
* **중요하다고 생각하는 것 (XML 파일 / 물체):** `dfs.datanode.failed.volumes.tolerated`
* **용법 파악:** 하나의 워커 노드에 여러개의 물리적 디스크를 설치하고 있을 때, 이 중 몇 개의 디스크에 장애가 생기면 노드를 Down으로 처리할 것인지 그 threshold를 설정한다.
* **중요하다고 생각하는 이유:** 기본값은 0으로 디스크가 1개만 고장나도 해당 노드는 전체가 일을 할 수 없는 것으로 처리된다. 값을 1로 세팅하면 하나의 디스크에 장애가 생기더라도 다른 정상적인 디스크 볼륨을 활용해 서비스를 지속하고, 클러스터의 고가용성을 높일 수 있다.

**팀원 B: 유용선**
* **중요하다고 생각하는 것 (XML 파일 / 물체):** `hdfs-site.xml` 파일의 `dfs.blocksize` 속성
* **용법 파악:** HDFS에 대용량 파일이 저장될 때 파일이 분할되는 기본 블록 단위의 크기를 지정하는 설정. 하둡은 파일 시스템에 들어오는 데이터를 이 크기 단위로 쪼개어 여러 데이터노드에 분산 저장하게 된다.
* **중요하다고 생각하는 이유:** 블록 크기 설정은 클러스터의 데이터 처리 성능과 안정성에 영향을 미치기 때문이다. 블록의 크기가 너무 작으면, 메타데이터의 용량이 커지고 맵리듀스 작업 시 맵 태스크가 너무 많아진다. 블록의 크기가 너무 크면, 태스크당 처리 시간이 길어지고 맵 태스크의 수가 줄어들어 분산 병렬 처리의 이점이 줄어든다.

**팀원 C: 이동찬**
* **중요하다고 생각하는 것 (XML 파일 / 물체):** `core-site.xml` 의 `fs.defaultFS`
* **용법 파악:** 클러스터의 기본 파일 시스템이 어디인지 지정하는 설정 파일로, 이 설정 덕분에 별도의 주소를 매번 쓰지 않아도 HDFS의 파일을 읽고 쓰며, Map-Reduce 작업이 올바른 분산 파일 시스템에서 실행된다.
* **중요하다고 생각하는 이유:** 애초에 `fs.defaultFS`가 틀려서 HDFS 자체에 연결이 안 되면 그 뒤 설정들은 아무 의미가 없음(업로드, 조회 등 실행 불가)

**팀원 D: 문태주**
* **중요하다고 생각하는 것:** `hdfs-site.xml` 파일 (`replications` 및 `dfs.blocksize` 속성)
* **용법 파악:**
  * replications: 얼마나 많은 복제본을 만들 것인지 지정
  * dfs.blocksize: HDFS의 블록 사이즈를 결정
  * 두개 모두 `hdfs-site.xml`에서 `<configuration>` → `<property>` 만들고 `<name>`과 `<value>` 태그를 사용해 정의한다.
* **중요하다고 생각하는 이유:**
  * replications: 같은 HDFS chunk에 대해서 얼마나 많은 복제본을 만들 것인지 체크할 수 있다. 많은 복제본을 만들 수록 가용성이 늘어나나, 중복 저장이 발생해 메모리 효용성이 낮아지는 trade-off가 중요하다고 생각한다. 하지만 병렬 처리에는 이는 지장은 없다. 중복해서 chunk로 저장해도 A worker와 B worker가 같은 chunk를 동시에 작업하지는 않는다.
  * dfs.blocksize: HDFS의 블록 사이즈를 결정한다. 블록 사이즈가 너무 작을 경우 map 테스크가 너무 커져서 불균형이 발생한다. 블록 사이즈가 너무 클 경우, 맵 태스크의 수가 줄어들어 worker 노드보다 chunk가 적어지면 병렬성이 사라진다.

## W3M4

### 1. 논의 주제

"'미리 정의된 키워드'가 아닌 다른 방법으로 나누고자 한다면 어떤 방법이 있을까"

### 2. 팀 토의 내용

* **제안된 '다른 방법' 아이디어:**

1. **VADER(가중치가 있는 감정 사전)**
    * **장점:** 별도의 모델 학습이 필요 없고 빠르다.
    * **단점:** 복잡한 문맥이나 뉘앙스를 파악하는 데는 한계가 있다.

2. **Bi-LSTM 모델 학습 (머신러닝 모델 적용)**
    * **장점:** 문맥까지 파악할 수 있어 정확도가 높다.
    * **단점:** 워커 노드가 가벼워야 한다는 하둡의 철학과 어긋난다. 
    * **대응법:**
        1. `mapper.py`에서 반복문 바깥에서 모델을 한번만 로드하고 재사용한다.
        2. 가벼운 모델을 선택한다.
        3. hadoop 말고 spark를 사용하면 task가 시작할때마다 모델을 로드 하는 방식이 아닌, 하나의 물리 컴퓨터 노드에서 executor에서 한번만 로드하고 이후에는 그 안에 있는 모든 task에 같은 메모리를 참조할 수 있다.

3. **이모티콘으로 긍정/부정 분류**
    * `:)`, `:-)`, `:D` 등이 포함되면 긍정, `:(`, `:-(` 등이 포함되면 부정으로 분류
    * **장점:** 규칙이 매우 직관적이며, 구현이 가장 쉽다.
    * **단점:** 이모티콘이 없는 텍스트의 감정은 파악할 수 없어 분석의 한계가 명확하다.
    
4. **LLM-judge**
    * **장점:** 정확도, 구현 쉬움, 별도 학습 필요 없음, 가장 뛰어난 성능
    * **단점:** 비용 문제, api를 사용한다고 했을때 네트워크 병목, 대용량 데이터에 대해서 map-reduce 적용하기 어려움(각 노드가 로컬에서 처리 불가능)
