# AV 유휴 여력 기반 도로신고 검증 데이터 프로덕트

> 매출을 만들지 못하는 자율주행 택시(AV)의 유휴 주행을, 도시의 미검증 311 포트홀 신고를 현장에서 확인·분류하는 공공 데이터 수집 채널로 전환하는 데이터 프로덕트

---

## 1. 개요

자율주행 택시(AV)의 매출이 발생하지 않는 유휴 시간을 활용하여, 아직 처리되지 않은 도시의 311 도로 신고(포트홀)를 현장에서 재확인하고 검증하는 데이터 프로덕트.

매출이 발생하지 않고 대기하거나 빈 차로 이동하는 시간에 자율주행 택시를 '공공 데이터 수집 모빌리티'로 전환함으로써, 도시는 검증 예산을 절감하고 운영사는 유휴 자산의 가치를 창출한다.

### 비즈니스 가치
도시는 검증을 위한 현장 출동 비용을 줄이고, 운영사는 매출이 0인 유휴 자산을 
1. **도시로부터의 검증 수수료**, 
2. **규제·라이선스 협상 크레딧**, 
3. **자사 노면 데이터 자산**이라는 세 갈래 가치로 전환한다.

---

## 2. 해결하는 문제

### ① 도시 측면
311로 포트홀이 접수돼도, 실제 조치 대상인지 확인하려면 사람이 현장을 봐야 한다. 뉴욕시의 경우 '조치 대상' 포트홀은 지름 1피트·깊이 3인치 이상으로 정의되며, 통상 DOT 검사관이 현장을 물리적으로 점검한 뒤 보수 여부를 결정한다. 이 검증 단계에 인건비·이동시간이 든다.

문제는 규모다. 뉴욕시 311은 통상 연간 6만 건 이상의 포트홀 관련 요청을 받고, DOT는 최근 연평균 약 17만 개의 포트홀을 보수한다. 여기에 중복·이미 보수됨·비(非)조치 대상(경미) 신고가 상당 비율 섞여 있어, 검증 자원의 일부는 "실제로는 처리할 것이 없는 곳"에 소모된다.

### ② 기존 모빌리티(사람 기사)의 구조적 한계 — 경제적 유인 부재
사람 택시 기사에게 "지나는 길에 저 좌표 노면 좀 확인해 달라"는 요청은 매출을 만들지 않는다. 검증 행위 자체에 보상 구조가 없으므로 응할 이유가 없고, 사람이 운전하는 차는 라이다·다중 카메라로 노면을 정밀 기록하지도 않는다. 결과적으로 현장 검증은 공공이 직접 인력을 투입해야만 하는 영역으로 남았다.

### ③ 자율주행 택시(AV) 운영사 측면 — 유휴 자산의 비효율
오프피크 시간대에 차량이 차고지에 서 있거나 빈 차로 배회하면 자산 활용도가 떨어진다. 이 시간은 운영비는 계속 발생하지만 매출은 0인 시간이다.

## 3. 타깃 대상

| 구분 | 타깃 고객/사용자 | 도입 목적 및 기대 역할 |
| :--- | :--- | :--- |
| **1차 타깃(주 사용자)** | **자율주행 택시(AV) 운영사**<br>(Waymo, Zoox(Amazon), Tesla, May Mobility 등 — 상업 운행 도시 우선) | • 유휴 여력이 발생하는 시간대/구역을 파악해 검증 우회(detour)를 배차 계획에 반영<br>• 도시 정부와의 우호적 파트너십 및 운행 라이선스 협상 레버리지 확보<br>• 자사 노면 데이터 자산 축적 |
| **2차 타깃(수혜자/구매자)** | **도시 교통국(DOT)**<br>(뉴욕시 DOT 및 공공 행정기관) | • 311 검증을 위한 현장 출동 예산·행정 인력 절감<br>• 실시간 검증 데이터로 보수 우선순위 최적화 및 시민 안전 제고<br>• 중복·해결·비조치 신고의 자동 종결로 백로그 축소 |

## 4. 비즈니스 모델(수익화)

| 모델 | 지불 주체 | 설명 |
| :--- | :--- | :--- |
| **검증 건당 수수료** | 도시 | 유효 검증 1건당 과금. 도시의 기존 현장 검증 단가보다 낮게 책정해 즉각적 비용 절감이 이루어지도록 함 |
| **규제 크레딧(비현금)** | 운영사 ↔ 도시 | 검증 기여를 라이선스·허가 협상 시 신뢰·안전 실적으로 환산. 현금 대신 규제 우호도로 정산. |
| **노면 데이터 자산** | 운영사(내부) | 축적된 고해상도 노면 데이터를 자사 매핑·주행 모델 고도화에 재사용(간접 가치). |

## 6. 분석 방법론

| 구분 | 내용 |
| :--- | :--- |
| **분석 대상 기간** | 2024년 3월 한 달 |
| **분석 범위** | 뉴욕시 5개 자치구 · 택시 존 263개 |
| **공급(유휴 AV) 데이터** | HVFHV(Uber/Lyft) 트립 기록 |
| **수요(헛걸음 신고) 데이터** | 311 신고 데이터 중 `descriptor = Pothole`, `status = Closed` 건 |
| **헛걸음 판정 기준** | 종결 사유(resolution) 텍스트에 "중복 신고 / 문제 미발견 / 이미 수리됨 / 기준 부합(비위험) / 소관 아님 / 타 부서 이관" 등의 문구가 포함된 경우 |
| **존내이동 시간 산출** | 각 zone에서 pickup·dropoff가 모두 발생한 트립들의 소요시간 중앙값(실측 기반) |
| **검증 소요시간 가정** | 건당 3분 |
| **소요 자원 계산 대상** | AV로 방지 가능(`coverable=1`)한 233건만 집계 |

> **참고 사항**
> - 3월은 겨울철 결빙·해빙 이후 포트홀 발생이 급증하는 시기로 알려져 있어, 오히려 검증 수요가 큰 달을 기준으로 분석했다. 즉 본 결과는 연중 평균보다 다소 보수적인(수요가 높은) 시나리오에 가깝다.
> - 공급 데이터로 옐로캡 대신 HVFHV를 사용한 이유: 옐로캡은 맨해튼 밖 운행이 드물어 5개 자치구 전역 분석에 부적합

### 검증시간 가정에 대한 민감도

| 검증시간(분) | 필요 차량-시간 | 상시 필요 대수 |
| :---: | :---: | :---: |
| 3.0 | 35.7h | 0.048대 |
| 5.0 | 43.4h | 0.058대 |
| 10.0 | 62.8h | 0.084대 |
| 20.0 | 101.7h | 0.137대 |

- 검증시간 가정을 최대 20분(3분의 약 7배)까지 늘려도 상시 필요 대수는 0.14대 미만이다.
- 즉 전용 차량 배치 없이도 유휴 AV만으로 충분히 커버 가능하며, 이 결론은 가정 변화에 강건하다.
- 역으로 계산하면, 상시 필요 대수가 전용 차량 1대에 도달하려면 건당 검증시간이 약 185분(3시간 이상)에 달해야 한다. 이는 현재 가정(3분)의 약 60배로, 실제 현장 검증 소요시간과는 거리가 먼 극단적 시나리오다.

### 시간대 패턴
- **0~2시, 21~23시** 등 심야·오프피크 시간대는 방지율이 100%에 근접한다.
- 반대로 **새벽 5~7시**는 신고가 몰리는데(방지율 13~23%) 이 시간대 유휴 공급이 상대적으로 부족해, 방지율이 하루 중 가장 낮다.

## 9. 시사점 및 한계

- **핵심 결론:** 2024년 3월 기준 헛걸음 신고의 46.6%를, 도시 전체 유휴 차량 자원의 0.0013% 미만으로 방지할 수 있었다. 상시 대기 전용 차량 없이도 기존 유휴 자원 재배치만으로 충분한 규모다.
- **우선 파일럿 지역:** Queens(신고량 최다·방지율 최저이므로 개선 여지가 최대이다), 이어서 방지율이 가장 높은 Bronx·Staten Island를 검증 파일럿 후보로 우선 검토.

### 한계
1. ‘잉여 차량’은 그 시각 그 존에 빈 차가 있었다는 이론에 근거한 계산이며, 실제 AV가 검증 임무를 수락·배차받을 수 있는지는 별도 검증이 필요하다.
2. 존내 이동시간만 반영했고, 존과 존 사이를 이동하는 시간은 포함하지 않았다. 실제로는 A존 신고 처리 후 B존으로 넘어가는 시간이 추가되므로, 위 수치 또한 이론에 근거한 계산이이서 현실과 약간의 차이가 있을 수 있다.
